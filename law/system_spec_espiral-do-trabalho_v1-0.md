## 1 — O que é a Espiral do Trabalho

A Espiral do Trabalho é o **Project Claude.ai** onde a operação profissional do Pentágono Rick acontece. É o ponto de coordenação cross-vertex — onde V2 Constellation, V3 Lab, V4 Yugar Commons venture, V5 Muda e o Hexágono Atlas Frames se encontram, se conectam, e geram receita, produto, comunidade e impacto.

A Espiral do Trabalho **não constrói código** — isso vive em Marco 0. A Espiral do Trabalho **não vive a vida** — isso vive na Espiral Pessoal. A Espiral do Trabalho **opera os vértices**.

### 1.1 Origem (re-framing de 12 Mai 2026)

Na sessão de 12 Mai 2026 (wrap `mineracao-topologia`), Rick propôs o re-framing fundamental:

> *"Separamos o dev em Marco 0, então o Pentágono inteiro vira Trabalho."*

Esse re-framing destravou 3 auto-resoluções em sequência:
- Constellation cluster (5 Projects) → V2 Trabalho
- Yugar Commons venture → V4 Trabalho
- MUDA → V5 Trabalho

Antes do re-framing, havia ambiguidade entre o que era dev e o que era operação. Após, a separação é cirúrgica.

### 1.2 Princípio fundamental

**Dev separado em Marco 0 → Pentágono inteiro = Trabalho.**

Marco 0 contém o protocolo (Atom Engine), o código (mindroot-v2 / app Atom), o schema, os ADRs técnicos e o roadmap de software. Trabalho contém a operação de cada vértice — clientes, parceiros, contratos, marketing, equipes, finanças, R&D aplicado.

A fronteira é binária: se o entregável é código, é Marco 0. Se o entregável é qualquer outra coisa (consultoria, produto físico, evento comunitário, conteúdo, decisão de negócio), é Trabalho.

## 3 — Mapeamento Pentágono × camadas operacionais

O Pentágono existe em **uma geometria**, mas opera em **três camadas** (Marco 0 / Espiral do Trabalho / Espiral Pessoal). Cada peça do Pentágono vive em pelo menos uma camada — algumas atravessam mais de uma.

| Peça | Marco 0 (dev) | Espiral do Trabalho (operação) | Espiral Pessoal (vida) |
|------|:---:|:---:|:---:|
| **Atom Engine** | ✅ vive aqui | (usa o produto) | (usa o produto) |
| **V1 MindRoot** | ✅ dev do app | ✅ gestão de produto, roadmap, brand | ✅ uso pessoal cotidiano |
| **V2 Constellation** | (dev quando virar add-on) | ✅ consultoria + sistema + brand | — |
| **V3 Lab** | — | ✅ R&D (Atlas Atom mora aqui) | — |
| **V4 Yugar Commons** | — | ✅ venture (Airbnb + grants + co-op) | ✅ terra + família + Mt Samson |
| **V5 Muda** | — | ✅ filosofia + caridade | — |
| **⬡ Atlas Frames** | — | ✅ empresa LGS (orbita V3 Lab) | — |

### 3.1 V1 MindRoot — peça que atravessa todas as camadas

V1 MindRoot é a única peça que vive nas **três** camadas simultaneamente. Não é problema — é a natureza do app:

- **Marco 0** constrói o app (dev contínuo do mindroot-v2)
- **Espiral do Trabalho** trata o app como produto a entregar (gestão, brand, roadmap, marketing)
- **Espiral Pessoal** trata o app como ferramenta de vida (Rick usa o Atom diariamente pra organizar a própria vida)

Cada camada toca V1 sob uma ótica diferente. O mesmo objeto, três funções.

### 3.2 V4 Yugar Commons — peça com split limpo

V4 também atravessa duas camadas, mas com **split limpo** (D-007 §4.4):

- **Espiral do Trabalho** = venture (Airbnb co-op + grants + parcerias comerciais)
- **Espiral Pessoal** = terra + subdivisão + hipoteca + família vivendo lá

O endereço físico (1874 Mount Samson Rd) é o mesmo — as funções operacionais são distintas. Trabalho trata o lado comercial, Pessoal trata o lado da vida.

### 3.3 Atlas Frames — Hexágono que orbita

Atlas Frames **não é vértice** — é o primeiro Hexágono ⬡ (Identidade v1.3 §2.7). Orbita o Pentágono e tessela infinitamente.

Operacionalmente, Atlas Frames orbita **V3 Lab** mais diretamente — porque o Lab faz R&D (ex: Atlas Atom modular system) que Atlas fabrica. Mas Atlas se conecta com o Pentágono inteiro:

- V1 MindRoot — Atlas usa o app pra organizar projetos e equipe
- V2 Constellation — Atlas é case de aplicação da metodologia Constellation
- V3 Lab — orbita primária (R&D → fabricação)
- V4 Yugar Commons — Atlas constrói no terreno
- V5 Muda — programas de impacto via Atlas

Atlas vive na Espiral do Trabalho como entidade operacional (empresa LGS/aço, Brendale QLD, time, Pipedrive, Xero, BAS).

## 4 — Os 5 vértices na Espiral do Trabalho

### 4.1 V1 MindRoot — gestão de produto

**Função operacional:** o app Atom como produto a entregar ao mundo.

**O que vive aqui:**
- Roadmap macro (espirais do app, fases de release)
- Brand e marketing
- Decisões de produto cross-funcionais
- Métricas de adoção (quando relevante)
- Documentação pública (quando relevante)

**O que NÃO vive aqui:**
- Código, dev, schema, migrations → Marco 0
- Uso pessoal do app pelo Rick → Espiral Pessoal

**Fronteira:** se a decisão é "como o produto se apresenta ao mundo," é aqui. Se é "como o produto funciona por dentro," é Marco 0.

### 4.2 V2 Constellation — consultoria, sistema, brand

**Função operacional:** o vértice de organizações. Inclui consultoria humana (Flora co-founder), o sistema metodológico, o (eventual) software Constellation, e o brand.

**O que vive aqui (consolidação pós-mineração):**
- Constellation System (metodologia, frameworks)
- Constellation OS (interface, brand)
- Conteúdo absorvido de Anthropic (arquivado, KB distribuiu) e Macro Vision (arquivado, KB distribuiu)
- Atlas Frames Ecossistema (frente Atlas dentro de Constellation, separada da operação empresa)
- Decisões de Flora como co-fundadora
- Clientes e contratos

**O que NÃO vive aqui:**
- Dev do (eventual) software Constellation → Marco 0 (quando virar add-on Atom)
- Operação Atlas Frames empresa (empresa em si) → vive como Hexágono próprio

**Modelo B2B futuro:** "Atom for Teams" (Identidade v1.3 §2.5).

### 4.3 V3 Lab — R&D

**Função operacional:** o espaço de experimentação. Onde ideias nascem, testam e se provam antes de virarem produto em qualquer outro vértice ou hexágono.

**Sinônimos canônicos:** Lab = Yugar Lab = AtlasLab (mesma coisa, contextos diferentes — D-007 §4.1).

**O que vive aqui:**
- Atlas Atom (sistema construtivo modular SHS 89mm) — 28+ ADRs, builders 3D, repo `atlas-atom`
- Outros R&D que surgir
- Workshop Shed Container (espaço físico do Lab, Mt Samson)
- Specs e ADRs de produtos R&D

**Princípio do Lab:** experimenta pra todos. Não pertence a vértice nenhum — alimenta todos.

**Fluxo R&D → Operação:** Lab desenvolve, vértice/hexágono apropriado adota. Ex: Atlas Atom (Lab) → fabricação (Atlas Frames hexágono).

### 4.4 V4 Yugar Commons venture

**Função operacional:** lado venture/comercial do terreno Mt Samson (1874 Mount Samson Rd).

**O que vive aqui:**
- Airbnb co-op (operação de aluguel)
- Grants e parcerias com government/NGOs
- Subdivisão fundiária (lado comercial)
- Eventos pagos no terreno
- Modelo de negócio comunitário

**O que NÃO vive aqui:**
- Terra como bem da família, hipoteca, vivência → Espiral Pessoal
- Daniel/purchase pending da fazenda vizinha (decisão pessoal) → Espiral Pessoal
- Filosofia comunitária → V5 Muda

**Split limpo:** D-007 §4.4.

### 4.5 V5 Muda — impacto e filosofia

**Função operacional:** retorno ao mundo. Caridade, educação, filosofia da economia de presença.

**O que vive aqui:**
- Filosofia "presença sobre produtividade"
- Programas de caridade
- Educação (incluindo a base filosófica de MindRoot Kids)
- Conteúdo público de impacto

**O que NÃO vive aqui:**
- MindRoot Kids como produto/app → ciclo Pessoal ↔ Marco 0
- Operação comercial (não vai pra Muda) → V2 ou V4

**Operação:** MUDA opera via Constellation (V2 fornece o sistema, V5 fornece o propósito).

## 5 — O Hexágono: Atlas Frames empresa

Atlas Frames é o **primeiro Hexágono ⬡** (Identidade v1.3 §2.7). Não é vértice — orbita.

### 5.1 O que é

Empresa LGS (Light Gauge Steel) operacional em Brendale, QLD. Pré-existia ao Atom (não nasceu de dentro pra fora). Quando o Pentágono se conecta com Atlas, a conexão interna gera efeito cascata no mundo externo — isso é propagação.

### 5.2 O que vive aqui

- Operação Atlas Frames empresa (Pipedrive, Xero, time, contratos)
- 14 pendentes operacionais identificados na sessão de 12 Mai (NDA Willi, Gleisson, cash reserve 15%, BAS, QBCC, PI, pricing review, PM tool, etc.)
- Decisões empresariais (André como sócio operacional)
- Site rsf-super-landing-page e materiais comerciais
- Customer relationships

### 5.3 Relação com o Pentágono

Atlas orbita V3 Lab primariamente (R&D → fabricação), mas se conecta com todos os vértices conforme descrito em §3.3.

## 6 — Fronteiras

### 6.1 vs Marco 0

**Marco 0 contém:**
- Atom Engine (protocolo, Genesis, schema, state machine)
- Dev do mindroot-v2 (codename interno do app Atom)
- Migrations, ADRs técnicos, type-schemas.json
- Roadmap técnico do app

**Espiral do Trabalho contém:**
- Tudo descrito em §3-5

**Fronteira binária:** entregável é código? → Marco 0. Caso contrário → Trabalho (ou Pessoal).

### 6.2 vs Espiral Pessoal

**Espiral Pessoal contém:**
- Mt Samson família (terra como casa, hipoteca, Fernanda, Bella, Ben)
- Arquivo pessoal (bio, memórias, histórico)
- MindRoot Kids (frente cíclica — nasce na Pessoal, vai pro Marco 0, volta)
- Uso pessoal do app Atom (V1 Espelho)

**Cruzamentos explícitos:**
- V1 MindRoot atravessa Pessoal (uso) e Trabalho (gestão)
- V4 Yugar Commons split: Pessoal (terra/família) vs Trabalho (venture)

**Fronteira:** vida sendo vivida = Pessoal. Vida sendo monetizada/organizada como projeto = Trabalho.

### 6.3 vs O Espaço Entre

**O Espaço Entre / Project E.** é standalone, protegido, fora do sistema (Meta-Template v1.2 §13). A Espiral do Trabalho **nunca toca, nunca referencia, nunca opera dentro** do Espaço Entre.

Se algo nasce no Espaço Entre e quer entrar no sistema operacional, **passa pelo inbox normal** (Genesis v5.0.4 Part 2, regra inbox obrigatório). Não há atalho via Trabalho.

## 7 — Princípios operacionais

### 7.1 Tagging cross-vertex

Items no Atom relacionados à Espiral do Trabalho usam tags:
- `#vertice:v1-espelho` — gestão V1
- `#vertice:v2-constellation`
- `#vertice:v3-lab`
- `#vertice:v4-yugar-commons`
- `#vertice:v5-muda`
- `#hexagono:atlas-frames`

Items podem ter múltiplas tags de vértice (cross-vertex). Ex: campanha de marketing de Atlas Atom = `#hexagono:atlas-frames` + `#vertice:v3-lab`.

### 7.2 Fluxo entre vértices

O Pentágono é fechado mas sempre aberto. Items podem migrar entre vértices via morph ou via connections tipadas (Genesis v5.0.4 Part 4).

Fluxos canônicos identificados:
- **V3 Lab → ⬡ Atlas Frames:** R&D vira produto fabricável
- **V2 Constellation ↔ V5 Muda:** filosofia ↔ metodologia ↔ programas de impacto
- **V4 Yugar Commons venture → V5 Muda:** lucro comunitário alimenta impacto
- **V1 MindRoot → todos:** app é ferramenta usada por todos os vértices

### 7.3 Marco 0 não escreve aqui (e vice-versa)

Espiral do Trabalho **não comita código**. Marco 0 **não opera negócio**. A separação é estrutural e protege os dois lados:

- Trabalho não fica refém de dev (operação não para se Marco 0 estiver em refactor)
- Marco 0 não fica refém de operação (dev não para se cliente reclamar)

Quando há ponto de contato, é via item (AtomItem) — Trabalho cria item no inbox descrevendo necessidade, Marco 0 prioriza no roadmap técnico.
